# 评估模块新手入门

本文档面向刚接触本仓库评估（evaluation）部分的新手开发者，帮助快速理解模块职责、目录结构和常用开发场景。

---

## 一、评估模块是做什么的？

评估模块负责：**在 Isaac Lab 仿真环境里跑你的 VLA 策略，并驱动「观测 → 策略 → 动作 → 环境 step」的闭环**。  
它不实现具体策略逻辑，只负责：

- 创建并管理仿真环境（通过任务配置）
- 把环境数据整理成**统一观测格式**（ObservationDict）给策略
- 从策略拿到**动作**后交给环境执行
- 可选：录制视频、支持关节/末端（EE）控制模式等

**数据流**：`环境` → `ObservationBuilder` → `ObservationDict` → `Policy` → `动作` → `环境.step()` → 下一帧观测。指标由环境在每次 episode 结束后自行记录。

## 二、目录结构一览

```
evaluation/
├── evaluator/           # 评估驱动
│   ├── vla_evaluator.py # 核心：VLA_Evaluator，持有 env + policy，跑 run_evaluation
│   └── VLAIsaacEnv.py   # 对 Isaac Lab 环境的薄封装（关节映射等）
├── observation/         # 观测构建
│   ├── schema.py        # 观测/动作的数据结构定义（ObservationDict、ActionDict 等）
│   └── builder.py       # ObservationBuilder：从 env 拼出 ObservationDict
├── models/policy/       # 策略实现（可扩展）
│   ├── base.py          # Policy 抽象接口，所有策略需继承
│   ├── random_policy.py
│   ├── openpi_remote_policy.py
│   ├── openvla_remote_policy.py
│   ├── curobo_reach_box_policy.py  # Curobo 规划 demo
│   └── ...
├── robot_registry.py    # 机器人注册表：robot_id → 臂自由度、Curobo 配置、相机键等
├── curobo/              # Curobo 运动规划封装（EE 模式用）
│   └── planner.py
└── tester/              # 测试框架：瞬移物体模拟抓放，验证环境指标
    └── tester.py
```

入口脚本在仓库根目录的 **`scripts/evaluate_vla.py`**，会创建 `VLA_Evaluator` 并调用 `run_evaluation()`。

## 三、核心概念（5 分钟速览）

| 概念 | 说明 |
|------|------|
| **环境 (Env)** | 由 `env_cfg`（如 `Spawn_ds_st_sparse_EnvCfg`）创建，继承自 Isaac Lab 的 `ManagerBasedRLEnv`。场景中的机器人、相机、任务逻辑都由任务配置决定。 |
| **观测 (Observation)** | 策略看到的不是 env 原始 group 观测，而是 **ObservationBuilder** 产出的 **ObservationDict**，包含 `meta`、`robot_state`、`vision`、`point_cloud` 等分组。 |
| **策略 (Policy)** | 实现 `observation/schema` 里约定的接口：接收 ObservationDict，返回动作 tensor（或带 `action` 的 dict）。可以是本地策略或远程服务（OpenPI/OpenVLA 等）。 |
| **评估器 (VLA_Evaluator)** | 持有一个 env 和一个 policy，在 `run_evaluation()` 里循环：reset → 取 obs → policy 得 action → step；可选录制视频、EE 模式下用 Curobo 做 IK。 |
| **机器人注册表 (robot_registry)** | 按 `robot_id` 提供评估侧配置：臂自由度、平台关节名、Curobo yml/URDF、相机/动作配置键等。**新机器人需在 REGISTRY 里注册一条**。 |

## 四、如何跑一次评估？

在项目根目录执行（需先配置好 `ASSET_ROOT_PATH` 对应的资产路径）：

```bash
python scripts/evaluate_vla.py \
  --task_scene_name Spawn_ds_st_sparse_EnvCfg \
  --policy random \
  --robot_id realman_dual_left_arm \
  --num_envs 2
```

**常用参数**：

- `--task_scene_name`：任务场景配置类名，与 `register.load_env_configs(...)` 使用的名称一致。
- `--policy`：策略名称，如 `random`、`openpi`、`openvla`、`curobo_reach_box` 等，见 `vla_evaluator.py` 里 `_make_policy_from_name`。
- `--robot_id`：评估侧机器人 ID，对应 `evaluation/robot_registry.py` 的 REGISTRY 键。
- `--num_envs`：并行环境数量（不写则用任务默认）。
- `--sim_device` / `--curobo_device`：仿真/渲染 GPU 与 Curobo 规划 GPU，如多卡时可设为 `cuda:0`、`cuda:1`。
- `--asset_root_path`：资产根路径，需与任务中 USD 等路径约定一致。
- `--from_json`：0=记录 JSON，1=回放 JSON，2=纯随机（默认）。

环境会在每次 episode 结束后自行记录指标；评估器只负责驱动仿真与策略调用。

## 五、如何接入自己的策略？

1. **实现策略类**  
   继承 `evaluation/models/policy/base.py` 里的 **`Policy`**，至少实现：
   - `name` 属性（字符串）
   - `predict(self, obs: Dict, **kwargs) -> torch.Tensor`  
     - 入参 `obs` 即 ObservationBuilder 产出的 **ObservationDict**
     - 返回 `(num_envs, action_dim)` 的 tensor，或返回带 `"action"` 键的 dict

2. **控制模式**  
   - 默认 `control_mode == "joint"`：动作直接为关节空间，交给 env.step。  
   - 若你的模型输出末端位姿（EE），可设 `control_mode == "ee"`，并在策略内部做 IK，或依赖评估器里集成的 Curobo 将 EE 目标转成关节动作（需在 `robot_registry` 中配置 Curobo 相关字段）。

3. **注册到评估器**  
   在 `evaluator/vla_evaluator.py` 的 **`_make_policy_from_name`** 中增加分支，例如：
   - 若用策略名 `"my_policy"` 且策略类为 `MyPolicy`，增加：
     ```python
     if policy_name == "my_policy":
         from ...models.policy.my_policy import MyPolicy
         return MyPolicy(...)
     ```
   这样命令行 `--policy my_policy` 就会创建你的策略实例。

## 六、如何接入新机器人？

1. **在 `evaluation/robot_registry.py` 的 REGISTRY 里加一条**  
   类型为 **`RobotEvalConfig`**，必填常用字段包括：
   - `robot_id`：与 `--robot_id` 一致
   - `arm_dof`：手臂关节数（如 7）
   - `platform_joint_name`：若有可动平台则填关节名，否则 `None`
   - `arm_base_offset_in_root`：臂基在根坐标系下的偏移
   - Curobo 相关（EE 模式需要）：`curobo_yml_name`、`curobo_asset_folder`、`curobo_urdf_name` 等
   - `camera_config_key` / `action_config_key`：与 `configs/` 下相机、动作配置对应

2. **配置文件**  
   若用 Curobo，需在 `configs/robot_configs/` 下提供对应 yml；资产在 `assets/robots/` 下按 `curobo_asset_folder` 等约定放置。

3. **使用**  
   命令行传入 `--robot_id 你注册的 robot_id` 即可。

## 七、观测与动作长什么样？（给写策略的同学）

- **观测**：`observation/schema.py` 中的 **ObservationDict**，顶层键包括：
  - `meta`：task_name、episode_id、step_id、num_envs 等
  - `robot_state`：qpos、qvel、qacc，以及可选的 root 位姿
  - `vision`：多相机 rgb、depth、seg、内参/外参等
  - `point_cloud`：可选，由 depth + 内参生成

- **动作**：通常为 **`(num_envs, action_dim)`** 的 tensor；若策略返回 dict，需包含 `"action"` 键。  
  `action_dim` 由环境的 `action_manager.total_action_dim` 决定，与 `robot_registry` 中该机器人的配置一致。

更细的 shape 与可选字段见 **`observation/schema.py`** 中的 TypedDict 定义。

---

## 八、其他

- **Tester**（`evaluation/tester/`）：通过瞬移物体模拟抓取/放置结果，用于验证环境的指标计算是否正确，而不是评估真实策略。需要时可看 `scripts/run_test.py` 的用法。
- **EE 模式与 Curobo**：若策略输出末端位姿，评估器可在 `_convert_actions_by_control_mode` 里调用 Curobo 做 IK，将 EE 目标转为关节动作；需在 `robot_registry` 中为该 `robot_id` 配置 Curobo 并保证 Curobo 环境可用。
- **视频**：评估器支持按步写视频，通过构造 `VLA_Evaluator` 时的 `record_video`、`video_output_dir` 控制。

遇到问题时，可先查：

1. `scripts/evaluate_vla.py` 的入参与 `VLA_Evaluator` 的构造参数  
2. `evaluation/robot_registry.py` 中当前 `robot_id` 的配置是否完整  
3. `evaluation/observation/schema.py` 中观测/动作的字段与 shape

祝开发顺利。
